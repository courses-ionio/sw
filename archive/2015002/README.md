## 1ο Παραδοτέο - Εισαγωγή

Μελλοντικά, στόχος μου είναι να δουλέψω ως Data Analyst ή Data engineer με Machine Learning και Deep Learning. Αυτός ο ρόλος απαιτεί να μπορείς να δουλέψεις ως μέλος μίας ομάδας όπου το κάθε άτομο ακολουθεί μεθοδικά οδηγίες για την υλοποίηση και λύση του εκάστοτε προβλήματος. Παράλληλα ως αναλυτής, πέρα από το team work, έχεις να κάνεις και με εξωτερικούς παράγοντες, όπως παρουσιάσεις τον ευρημάτων σου σε άλλα άτομα, συζήτηση με άλλες ομάδες κλπ.

Για τους πιο πάνω λόγους, ένας από τους βασικούς στόχους που έχω για το μάθημα είναι να μπορώ να συνεργαστώ με του συμφοιτητές μου, να παίρνω και να δίνω feedback για την δουλεία μου και παράλληλα να βελτιώνομε πάνω σε αυτό το feedback.

Αναμφισβήτητα η πλατφόρμα GitHub είναι ένα εξαιρετικά χρήσιμο και απαραίτητο εργαλείο. Για αυτό τον λόγο, με χαροποιεί ιδιαίτερα που θα την ξανά χρησιμοποιήσω για αυτό το μάθημα και για μελλοντικά πρότζεκτ όπως η [πτυχιακή μου εργασία!](https://github.com/mariosconsta/thesis)

Πέρα από τις δεξιότητες που θα πάρω από το μάθημα στο κομμάτι του Linux, θεωρώ πως το πιο σημαντικό για εμένα είναι το exposure που θα πάρουμε ως μαθητές για το πως είναι να δουλεύει κάποιος μέσα σε μία ομάδα όπου οι πράξεις όλων, επηρεάζουν την ομάδα σαν σύνολο. Στο περασμένο μάθημα δεν υπήρχαν ομάδες, η δουλειά που έβγαζες, επηρέαζε μόνο εσένα, θετικά ή αρνητικά. Τώρα ήμαστε σε μία ομάδα όπου όλοι πρέπει να συμβάλουμε στην υλοποίηση των παραδοτέων. Αυτό απαιτεί συντονισμό, οργάνωση και θέληση και παράλληλα υπάρχουν οι επιπτώσεις σε σχέση με την δουλειά που βγάζουμε.

Προσωπικά για εμένα, ο μεγαλύτερος στόχος που έχω για το μάθημα είναι να μάθω να δουλεύω σε μία ομάδα, να γίνω familiar με τα stages και steps που πρέπει να ακολουθήσουμε σαν ομάδα για την υλοποίηση ενός στόχου/λειτουργίας.
Τέλος, ο κύριος στόχος μου σε αυτό το μάθημα είναι να μάθω τι ακριβώς είναι ένα λογισμικό. Πέρα από τον κώδικα, υπάρχουν έννοιες, ιδέες και γενικότερα μία φιλοσοφία. Θεωρώ πως αυτή η φιλοσοφία ξεκίνησε να χάνετε. Πλέον τα πράματα έγιναν βατά, με εργαλεία που πλέον κάνουν 90% της δουλείας με αποτέλεσμα να μην χρειάζεται ο χρήστης / προγραμματιστής να καινοτομήσει ή να σκεφτεί διάφορους τρόπους επίλυσης προβλημάτων.

“The best way to predict the future is to invent it.” – Alan Kay. Μέσα από τα βίντεο του Alan Kay κυρίως, αλλά και των άλλων, κατάλαβα πως τα πράγματα που θεωρούμε εμείς δεδομένα και “standard” τώρα, τότε ήταν το Definition της καινοτομίας και του διαφορετικού τρόπου σκέψης. 

Τώρα μας δίνονται η λύσεις και όλα τα λογισμικά που θα χρειαστούμε ποτέ έτυμα. Το path που μαθαίνει κάποιος software development έχει γίνει “linear” κατά την άποψη μου, χωρίς να θέλω να ισοπεδώσω τους υπερβολικά ταλαντούχους Developers που έχουμε τώρα. Αυτό που θέλω να πω, είναι πως σαν μάζα, έχουμε σταματήσει να προσπαθούμε να δημιουργήσουμε και το learning path μας προορίζεται από το ίδρυμα φοίτησής μας.

Αυτό μου φαίνεται και περίεργο διότι, η πληροφορία που υπάρχει τώρα στο διαδίκτυο ή σε βιβλία αυξήθηκε με ραγδαίο βαθμό. Άρα, έχει πολλούς τρόπους κάποιος να αντλήσει πληροφορία και να μάθει.

Για να μην τα πολυλογώ, ο σκοπός μου σε αυτό το μάθημα είναι να μάθω να σκέφτομαι “outside of the box” και να μην κάνω settle με έτυμες λύσεις. Σκοπός μου είναι να αυξήσω την αυτοπεποίθηση μου στο να δοκιμάζω διάφορους τρόπους και τεχνικές για την δημιουργία λογισμικού και ας μου πάρει περισσότερο χρόνο σε σχέση αν έπαιρνα έτυμο κώδικα. 

Μία μικρή σημείωση που θέλω να κάνω είναι πως ανυπομωνώ να χρησιμοποιείσω αυτά που έμαθα στο μάθημα Επικοινωνία Ανθρώπου Υπολογιστή σε αυτό το μάθημα, όπως Scripting για παράδειγμα.

### What did I learn?

Αυτό το παραδοτέο μου έδειξε ένα διαφορετικό τρόπο που μπορεί κάποιος να έχει το βιογραφικό του. Εγώ μέχρι τώρα έχω απλά ένα αρχείο στον υπολογιστή μου και όποτε χρειάζεται αλλαγές, τις κάνω.

Βέβαια ένα βιογραφικό δεν είναι ποτέ static. Πάντα αλλάζει ανάλογα με την θέση που κάνεις Apply. Για παράδειγμα, αν εγώ κάνω Apply σε μία θέση που αφορά το IT κομμάτι, τότε το βιογραφικό θα πρέπει να εστιάζει σε δεξιότητες και γνώσεις που είναι σχετικές με IT. Όπως Hardware, Databases, Servers, κλπ. Τα υπόλοιπα που δεν είναι σχετικά με IT θα πρέπει να διαγραφούν. Αυτό το λέω από προσωπική μου εμπειρία όπου έκανα Apply σε μία δουλεία σχετική με το IT κομμάτι και επειδή δεν είχα αρκετή προϋπηρεσία, έβαλα και την δουλεία που έκανα ως Barista σε ένα καφέ. Την δουλεία την κέρδισα αλλά το feedback που έλαβα για το βιογραφικό μου ήταν αυτό που έγραψα πιο πάνω.

Στο μέλλον, σκοπεύω να κάνω apply για το πόστο του Machine Learning Engineer. Το βιογραφικό θα πρέπει να αλλάξει. Θα μπουν γνώσεις και δεξιότητες που αφορούν το πόστο που κάνω apply.

Εκεί που θέλω να καταλήξω είναι πως με τον τρόπο που έμαθα στο μάθημα, είναι πως μπορώ να έχω διάφορα “versions” του βιογραφικού, για διάφορες θέσεις εργασίας που θα κάνω apply και με αυτοματοποιημένο τρόπο να γίνονται όλα Updated.

Με αυτό τον τρόπο έχω ένα βιογραφικό για κάθε θέση εργασίας που ενδιαφέρομε να εργαστώ και παράλληλα δίνεις το option στο άτομο που σε κάνει recruit να δει ακριβώς αυτά που τον ενδιαφέρουν με αποτέλεσμα να αφήσεις μία καλή εντύπωση και να αυξήσεις τις πιθανότητες στο να σε πάρουν.

Τέλος, μέσα από ένα τέτοιο βιογραφικό, δείχνεις πως έχεις γνώσεις σε Version Control Software και πως σαν άτομο είσαι efficient. Είμαι πολύ ευχαριστημένος και σίγουρα στο δικό μου χρόνο, θα το εμπλουτίσω περισσότερο.

### Δυσκολίες

Σε αυτό το παραδοτέο δυσκολεύτηκα πιο πολύ στο κομμάτι του Pull Request. Υπήρχε ένα μικρό μπέρδεμα για τον τρόπο που έπρεπε να κάνουμε το Pull Request. Αρχικά, έκανα τις αλλαγές και άνοιξα Pull Request στο sitegr. Εν τέλη όμως, οι αλλαγές που έκανα, ανήκαν στο submodule all_collections. 

Εγώ δεν ήξερα πως έπρεπε να ανοίξω Pull Request στο all_collections και μετά από αρκετό διάβασμα συζητήσεων που είχε ο καθηγητής για το ίδιο πρόβλημα, κατάλαβα τι έπρεπε να κάνω, και επίσης κατάλαβα γιατί ήταν λάθος το αρχικό Pull Request, όπου φαινότανε απλά το pointer του Submodule αντί οι αλλαγές μου. 

Επίσης έκανα πολλές αλλαγές αφού άνοιξα το Pull Request και για αυτό γέμισε με άσχετα commits τα οποία έπρεπε να τα κάνω “Squash”, κάτι που δεν ήξερα πως μπορώ να κάνω. Είχα χαθεί μέσα στα commit και έγινε ένα χάος με την διαδικασία του Rebase. 

Είμαι σίγουρος πως η διαδικασία ήταν πολύ απλή, οι λίγες γνώσεις που είχα στο κομμάτι αυτό ήταν η αιτία που έγιναν όλα πολύπλοκα. Το καλό είναι πως έμαθα πως δουλεύει το Rebase/Revert, έμαθα για τα Pointers των Submodules και έμαθα πως να ανοίγω διάφορα Pull Requests χωρίς Conflicts και άσχετα Commits! A clean Pull Request is a happy Pull Request!

Τέλος θα ήθελα να σχολιάσω πως μου άρεσε που μπορεί κάποιος να βάλει Closes #3 και αμέσως το GitHub ξέρει για πιο issue / PR μιλάει ο χρήστης και τα κλίνει αυτόματα. Μου φάνηκε πολύ “clean”!

### What did I learn?

Εδώ έμαθα πως να κάνω Squash τα πολλά και αχρείαστα Commits σε ένα Pull Request με την μέθοδο Rebase. Έμαθα να χειρίζομαι και να διαβάσω το Branch Tree στο Command Line και να διαλέγω ποια Commits να κρατήσω. Στην τελική αναφορά και Pull Request του μαθήματος θα το ξανά κάνω έτσι ώστε το Pull Request να μην έχει πολλά Commits αν προκύψουν.

## 4o Παραδοτέο - Άσκηση γραμμής εντολών

Για το 4ο παραδοτέο αποφάσισα να εγκαταστήσω το GhostBSD, το οποίο είναι βασισμένο πάνω στο FreeBSD, μέσο VirtualBox. Κοίταξα διάφορες επιλογές για λογισμικά και αποφάσισα να πάω με το GhostBSD διότι μου φάνηκε απλό και λειτουργικό. Με την πάροδο του χρόνου ίσος δοκιμάσω και άλλα, αλλά για την ώρα θα παίξω με αυτό.

### What did I learn?

Εδώ η διαδικασία ήταν ήδη γνώστη σε εμένα και αρκετά απλή αφού έγινε χρήση του VirtualBox. Βέβαια, για προσωπική χρήση, είχα σκοπό να εγκαταστήσω με Dual Boot ένα Linux Distro, ίσος Ubuntu διότι υπάρχουν Drivers για την κάρτα γραφικών μου που δεν είναι διαθέσιμοι στα Windows. Οι Drivers αυτοί μου επιτρέπουν να χρησιμοποιήσω την κάρτα γραφικών για deep learning model training, κάνοντας το training υπερβολικά ποιο γρήγορο.

Η πλατφόρμα που χρησιμοποιώ λέγετε Tensorflow και αν έχεις κάρτα γραφικών της nVidia, μπορείς να εκπαιδεύσεις το μοντέλο με την κάρτα. Εγώ έχω κάρτα της AMD με αποτέλεσμα να μην μπορώ. Η AMD βέβαια, έκανε release drivers που σου επιτρέπει να κάνεις training με AMD GPU αλλά τα drivers είναι διαθέσιμα σε linux. Για αυτό τον λόγο, σε αυτό το παραδοτέο μελέτησα αρκετά Linux Distributions και έχω μερικά στο μάτι μου για τον σκοπό που ανάφερα ποιο πάνω.

### Δυσκολίες

Σε αυτό το παραδοτέο είχαμε και την πρώτη πρόκληση σαν ομάδα αλλά και προσωπικά. Αρχικά πάντα δουλεύαμε μόνη μας, δεν υπήρχαν ποτέ conflicts (στις ασκήσεις συμμετοχικού υλικού) και έκανεs ότι αλλαγή ήθελες χωρίς να πρέπει να το συζητήσεις με κάποιον άλλο. Επίσης συνειδητοποιήσαμε πως η ομάδα δεν είναι λειτουργική έτσι όπως είναι. Είπαμε αρχικά να υπάρχει “αναρχία”. Όλοι είναι ίση. 

Αυτό το παραδοτέο μας απέδειξε πως δεν είναι εφικτό να λειτουργεί μία ομάδα με τέτοιο τρόπο. Τα πρώτα δύο Pull Request μας έβγαλαν αμέσως Conflicts. Τα κάναμε Revert. Στην συνέχεια έκανα reset το δικό μου Forked Repo και ξανά έκανα την διαδικασία απλά με μία μικρή διαφορά.

Δημιούργησα δύο καινούρια Branches. Το ένα το ονόμασα Netlify όπου εκεί άλλαξα τα αρχεία .gitmodules και config έτσι ώστε να δείχνουν στο προσωπικό μου site. Με αυτό τον τρόπο, τα αρχεία στο Master Branch δεν άλλαξαν. Στην συνέχεια δημιούργησα ακόμη ένα Branch το οποίο ονόμασα A1A2 – 2015002 όπου σε αυτό το branch έκανα τις αλλαγές που αφορούσαν τα παραδοτέα Α1Α2, δεν άλλαξα απολύτως τίποτα άλλο. Επίσης , άνοιξα Α1Α2 – 2015002 branches και στα ανάλογα submodules, images και _gallery.

Όταν πήγα να κάνω ξανά Pull Request, αυτή την φορά από το A1A2 – 2015002 branch, δεν είχα κανένα conflict και τα αρχεία που άλλαζαν ήταν μόνο το υλικό που ήθελα να προσθέσω για το παραδοτέο. Επίσης άλλαξα τις ρυθμίσεις στο Netlify, για να κάνει build **ΜΟΝΟ** από το Netlify Branch.

Μετά το επιτυχημένο merge του Pull Request, έκανα merge το Α1Α2 – 2015002 στο Netlify Branch για να φαίνονται οι αλλαγές και στο δικό μου Netlify.

Αν ένα άτομο βρήκε τόσα εμπόδια για το πρώτο ομαδικό παραδοτέο, τότε καταλάβαμε πως αν δεν θέσουμε κάποιους κανόνες και μία ροή που θα πρέπει όλοι να δουλεύουμε, τότε θα είχαμε τεράστιο πρόβλημα σαν ομάδα. Η αναρχία δεν λειτουργεί!!

Για αυτό τον λόγο αποφασίσαμε πως εγώ μαζί με άλλα δύο παιδία, να γίνουμε συντονιστές και είπαμε να γράψουμε μερικούς κανόνες για το πως θα πρέπει να γίνεται η κάθε διαδικασία μέσα στην ομάδα.

#### Τα δύο script που χρειάστηκαν για την ανανέωση του PDF με κάθε αλλαγή:

***post-commit***

```
#!/bin/sh
echo
if [ -e .commit ]
    then
    rm .commit
    cd pdf
    pandoc https://mariosconsta.github.io/cv/ -o cv.tex -s
    pdflatex --interaction=nonstopmode cv.tex
    git add .
    git commit --amend -C HEAD --no-verify
    ntfy -b telegram send "CV in PDF format updated"
fi
exit
```

***pre-commit***

```
#!/bin/sh
echo
touch .commit
exit
```
